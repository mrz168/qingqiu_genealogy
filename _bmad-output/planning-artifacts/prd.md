# Product Requirements Document - qingqiu_genealogy

**Author:** BobCheung
**Date:** 2026-03-06

## Executive Summary

**目标用户：** 有家族认同感的家族，希望将纸质族谱电子化，并以多格式（文字、图片、视频）记录家族成员信息，让每位成员的根系永久保存并更丰富。

**问题：** 纸质族谱仅能记录基本信息，易丢失、损坏，难以传承；后辈若无法看到先祖的面貌、经历与性格，难以建立真实感。市面族谱产品多为 SaaS 收费模式，数据难以自主掌控，且缺乏开源、可自部署的选择。

**愿景：** 让族谱从「名字+生卒年」变成「可看见、可感受的人生故事」——后辈能像看电影一样了解先祖的一生：看到太奶长什么样、经历何事、一生如何度过。系统面向电脑端（响应式）、手机 APP、小程序等多端，让每位成员的信息可被永久保存与传承。

### What Makes This Special

- **开源可自部署**：代码开源，可自部署、可二次开发，不受 SaaS 厂商限制；数据存储在家族自有环境，归属清晰、隐私可控。
- **多媒体记录**：文字 + 图片 + 视频，让后辈看到先祖的真实面貌与经历，而非仅名字与生卒年。
- **全平台覆盖**：电脑、手机 APP、小程序，满足不同场景与使用习惯。
- **核心洞察**：若古代有类似系统，今人可知道秦始皇、李世民等人的长相与经历。族谱不只是名字，而是可感知的人物与故事。

## Success Criteria

### User Success

- **核心体验达成**：用户能完成「像看电影一样了解先祖」——查看某位先祖的照片、生平、故事与关系，形成完整人物印象。
- **数据录入顺畅**：管理员能在 30 分钟内完成至少 50 名成员的批量导入，并自动建立父子、配偶等关系。
- **关系网可浏览**：用户能在树状图或世系表中浏览家族关系，支持缩放、拖拽、搜索与定位。
- **审批流程闭环**：成员提交生平修改后，审批人在 7 天内完成审批，申请人能收到结果通知。

### Business Success

- **开源项目**：以社区与采用为主，关注部署数量、GitHub Star、社区贡献与反馈。
- **3 个月**：完成 MVP 发布，至少 1 个家族完成部署并持续使用。
- **12 个月**：社区初具规模，有外部贡献者参与，文档与部署指南完善。

### Technical Success

- **可部署**：支持 Docker 一键部署，健康检查接口正常。
- **数据规模**：支持万级成员的关系网生成与展示，为百万级预留异步更新能力。
- **数据安全**：支持操作日志、定期备份与数据导出。

## User Journeys

### 旅程 1：家族成员 — 了解先祖（成功路径）

**人物：** 小陈，28 岁，在外地工作，对家族历史感兴趣，但从未见过太奶。

**Opening Scene：** 小陈听父亲提起太奶的故事，但家里只有一张模糊老照片和简单生卒年，无法想象太奶的样子和一生。

**Rising Action：** 父亲告诉他家族已把族谱搬到线上。小陈登录系统，在树状图中找到太奶，点击进入她的详情页。看到多张照片、生平故事、迁居记录和重大事件时间线。

**Climax：** 小陈第一次「看见」太奶：她的相貌、经历、性格，像看电影一样串联起来。他意识到自己与这位先祖有了真实的情感连接。

**Resolution：** 小陈把链接分享给妹妹，两人一起浏览家族树，讨论各自对先祖的理解。家族记忆在数字世界里延续。

---

### 旅程 2：家族成员 — 补充自己的生平（含审批）

**人物：** 李阿姨，55 岁，退休教师，希望把自己的经历写进族谱。

**Opening Scene：** 李阿姨发现族谱里自己的信息只有姓名和出生日期，想补充职业、重要经历和几张照片。

**Rising Action：** 她在自己的成员卡片上点击「编辑生平」，用富文本写下从教经历、获奖和家庭故事，并上传几张照片。提交后系统提示「已提交审批」。

**Climax：** 族长在待办中心看到李阿姨的修改申请，审阅后通过。李阿姨收到站内信：「您的生平修改已通过，现已生效。」

**Resolution：** 李阿姨刷新页面，看到自己的生平已更新。她感到自己的故事被家族正式记录，对族谱的参与感更强。

---

### 旅程 3：管理员 — 批量导入族谱

**人物：** 老王，族长，负责本次修谱，手头有 Excel 整理的 200 人名单。

**Opening Scene：** 老王需要把 200 人的信息录入系统，手工逐条录入不现实。

**Rising Action：** 他下载系统提供的 Excel 模板，按模板填写姓名、性别、出生日期、父亲姓名、母亲姓名等。上传文件后，系统进行校验并展示预览和错误报告。他修正 3 处格式错误后重新上传。

**Climax：** 系统根据「父亲姓名」「母亲姓名」自动建立父子、母子关系，并生成关系网。老王在树状图中看到完整的家族结构。

**Resolution：** 老王在 30 分钟内完成 200 人导入，关系自动建立。他邀请几位族人登录查看，修谱效率大幅提升。

---

### 旅程 4：子管理员 — 维护本支数据（Phase 2）

**人物：** 阿芳，长房分支管理员，负责本房 80 人的数据维护。

**Opening Scene：** 长房有几位成员信息需要更新，主管理员希望由阿芳负责本支的日常维护与初审。

**Rising Action：** 阿芳登录后，在「我的管理范围」中看到长房成员列表。她收到成员提交的生平修改申请，先做格式与内容初审，通过后提交给族长终审。

**Climax：** 阿芳发现某成员填写的父亲姓名与现有成员不匹配，她在驳回意见中说明原因，成员修正后重新提交。

**Resolution：** 长房数据保持准确，主管理员无需处理大量琐碎申请，审批流程更顺畅。

## Domain-Specific Requirements

族谱领域的合规、技术约束与风险应对。

### Compliance & Regulatory

- **数据归属**：数据存储在家族自有环境，需明确备份、导出与迁移能力，满足家族对数据掌控的预期。
- **操作审计**：关键操作（成员增删改、关系变更、审批）需记录操作日志，便于追溯与纠错。

### Technical Constraints

- **关系完整性**：建立/修改关系时需做循环检测（如 A→B→C→A），发现环路则拒绝并提示。
- **冲突检测**：父子、配偶等关系需做冲突校验，避免逻辑矛盾（如 A 既是 B 的父亲又是 B 的儿子）。
- **日期表达**：支持精确日期、约略日期（年/月/日精度）、朝代、不详等多种形式，并在展示时统一规范。

### Integration Requirements

- **MVP 阶段**：以 Excel 导入为主，暂不要求对接 GEDCOM 等外部族谱格式。
- **后期**：可预留 GEDCOM 导出，便于与其他族谱软件互通。

### Risk Mitigations

| 风险 | 应对 |
|------|------|
| 关系循环引用 | 建立/修改关系时做环路检测，拒绝并提示 |
| 导入数据质量差（重名、格式混乱） | 模板校验、格式规范、错误报告、支持修正后重导 |
| 审批长期未处理 | 待办提醒、超时提醒，后期可支持委托与转交 |
| 大规模关系网更新耗时 | 小规模同步，大规模预留异步队列与增量更新 |

## Innovation & Novel Patterns

产品差异化与创新点分析。

### Detected Innovation Areas

- **开源自部署模式**：在族谱领域提供可自部署、可二次开发的开源方案，数据由家族自有环境掌控，区别于主流 SaaS 模式。
- **多媒体叙事式族谱**：以照片、视频、生平故事构建人物叙事，让后辈「像看电影一样」了解先祖，而非仅查看结构化数据。
- **中国家族关系建模**：支持父子、配偶、收养、过继、多配偶等关系类型，并做冲突与循环检测，适配中国家族文化场景。

### Market Context & Competitive Landscape

- **市场现状**：族谱产品多为 SaaS 付费，数据托管在第三方，缺乏开源、可自部署选项。
- **差异化**：开源 + 自部署 + 多媒体叙事，在家族文化传承领域形成独特定位。

### Validation Approach

- **MVP 验证**：至少 1 个家族完成部署并持续使用 3 个月以上。
- **体验验证**：用户能完成「像看电影一样了解先祖」的核心路径。
- **社区验证**：12 个月内出现外部贡献者或社区反馈。

### Risk Mitigation

| 风险 | 应对 |
|------|------|
| 开源社区采用不足 | 优先满足单家族自用场景，不依赖社区规模 |
| 多媒体叙事体验不达预期 | 先保证基础树状图与世系表，再迭代叙事与时间线 |
| 自部署门槛高 | 提供 Docker 一键部署与清晰部署文档 |

## Web App Specific Requirements

### Project-Type Overview

qingqiu_genealogy 以 **Web 应用** 为主形态，采用 SPA 架构，支持响应式布局，兼顾电脑与平板。后期计划扩展小程序与 App。MVP 阶段聚焦 Web 端。

### Technical Architecture Considerations

- **前后端分离**：Vue 3 + Vite 前端，Django-Ninja RESTful API 后端
- **家族树可视化**：AntV G6 实现树状图，支持大规模图、懒加载、缩放拖拽
- **富文本编辑**：TipTap / Quill 用于生平编辑（富文本 + Markdown）
- **文件存储**：django-storages，支持本地存储与对象存储（S3/OSS/MinIO）

|--------|----------|------|
| Chrome | 最近 2 个主版本 | 主要目标 |
| Firefox | 最近 2 个主版本 | 支持 |

- **MVP**：以桌面与平板为主，手机端后期优化
- **树状图**：桌面全功能；平板/手机支持基础浏览，复杂交互可简化

### Performance Targets

详见 [Non-Functional Requirements - Performance](#performance)。

### SEO Strategy

- **MVP**：族谱为登录后内容，SEO 优先级低
- **后期**：若有公开家族页，可考虑基础 SEO（meta、结构化数据）

### Accessibility Level

- **目标**：WCAG 2.1 AA
- **MVP**：键盘导航、焦点管理、基础语义化
- **后期**：移动端适老化（大字体、简化操作）

### Implementation Considerations

- **API 版本**：`/api/v1/` 前缀，便于后续演进
- **认证**：Session 或 JWT，与 Django-Ninja 兼容
- **文档**：OpenAPI/Swagger（Django-Ninja 内置）

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP 思路：** 问题解决型 MVP（Problem-Solving MVP）——优先解决「纸质族谱难传承、市面 SaaS 数据不可控」的核心问题，让至少 1 个家族完成部署并持续使用。

**资源要求：** 1–2 名全栈开发，熟悉 Django + Vue 3；可先做单家族自用验证，再逐步扩展。

### MVP Feature Set（Phase 1）

**支持的核心用户旅程：**

- 家族成员：查看族谱、了解先祖（照片、生平、故事）
- 家族成员：修改自己的生平并提交审批
- 管理员：批量导入 Excel、建立关系
- 审批人：审批生平修改申请

**Must-Have 能力：**

| 能力 | 说明 |
|------|------|
| 成员 CRUD | 基本信息、生平、照片 |
| 关系建立 | 父子、配偶 + 冲突与循环检测 |
| 关系网生成 | 小规模同步，预留异步接口 |
| 树状图 + 世系表 | 基础展示 |
| 生平修改审批 | 站内信通知 |
| Excel 批量导入 | 预览、校验、错误报告 |
| 用户认证 | 手机/邮箱 |
| Docker 部署 | 健康检查 |

### Post-MVP Features

**Phase 2（Growth）：**

- 子管理员 + 权限范围
- 邀请制 + 审批
- 社交登录（微信、QQ）
- 懒加载、聚焦当前用户、缩放拖拽
- 多通道通知（邮件、短信、微信）
- 关系网异步更新（Celery + 增量）
- 导出（世系表/树状图/名录、JSON）

**Phase 3（Expansion）：**

- 纪念日提醒、家族公告
- 向上追溯视图
- 统一待办中心
- 移动端（小程序、App、适老化）
- 族谱年报、家族故事集

### Risk Mitigation Strategy

| 风险类型 | 应对策略 |
|----------|----------|
| **技术风险** | 关系网百万级采用异步+增量；小规模先同步实现，验证后再扩展 |
| **市场风险** | 以 1 个家族 3 个月持续使用为验证目标；开源社区采用不足时优先满足单家族自用 |
| **资源风险** | 最小团队 1 人；可先砍掉子管理员、邀请制等，仅保留管理员 + 成员 + 审批 |

## Functional Requirements

以下 FR 构成**能力契约**：UX 设计、架构与开发均需覆盖。未列出的能力将不会进入最终产品。

### 用户与认证

- FR1: 用户可以使用手机号或邮箱注册并登录系统
- FR2: 管理员可以创建族谱并成为该族谱的主管理员
- FR3: 系统可以为每个家族成员分配唯一账号标识（UUID）
- FR4: 已登录用户可以登出系统

### 成员管理

- FR5: 管理员可以添加家族成员，录入姓名、性别、出生日期、逝世日期、籍贯、生平、照片
- FR6: 管理员可以修改任意家族成员的信息
- FR7: 管理员可以删除无关联关系的家族成员
- FR8: 家族成员可以查看自己的成员详情
- FR9: 家族成员可以提交对自己生平信息的修改申请
- FR10: 系统可以支持不精确的出生/逝世日期（如「约 1900 年」「清末」「不详」）

### 关系管理

- FR11: 管理员可以建立成员之间的父子、母子、配偶关系
- FR12: 系统可以在建立或修改关系时检测并拒绝循环引用（如 A→B→C→A）
- FR13: 系统可以在建立或修改关系时检测并拒绝关系冲突（如 A 既是 B 的父亲又是 B 的儿子）
- FR14: 系统可以根据已录入的关系自动生成家族关系网
- FR15: 管理员可以在树状图上通过点选父/母/配偶来建立关系

### 族谱展示

- FR16: 已登录用户可以以树状图形式浏览家族关系
- FR17: 已登录用户可以以世系表形式浏览家族关系
- FR18: 已登录用户可以在树状图或世系表中搜索并定位成员
- FR19: 已登录用户可以点击成员查看其详情（照片、生平、关系、时间线等）
- FR20: 已登录用户可以分享成员或族谱的查看链接给其他族人
- FR21: 系统支持树状图的缩放与拖拽操作

### 审批流程

- FR22: 家族成员提交生平修改后，系统可以自动发起审批流程
- FR23: 审批人（管理员/族长）可以查看待审批的生平修改申请
- FR24: 审批人可以通过或驳回生平修改申请
- FR25: 审批人驳回时可以提供驳回意见
- FR26: 申请人可以在申请被驳回后修改内容并重新提交
- FR27: 审批通过后，系统可以将修改后的生平生效并通知申请人
- FR28: 申请人可以通过站内信接收审批结果通知

### 批量导入

- FR29: 管理员可以下载 Excel 导入模板
- FR30: 管理员可以上传 Excel 文件进行批量成员导入
- FR31: 系统可以在导入前对数据进行校验并展示错误报告
- FR32: 系统可以根据 Excel 中的「父亲姓名」「母亲姓名」自动建立父子、母子关系
- FR33: 管理员可以在确认预览后执行导入
- FR34: 导入完成后，系统可以触发关系网更新

### 系统管理

- FR35: 管理员可以查看操作日志（成员增删改、关系变更、审批等）
- FR36: 管理员可以执行数据备份
- FR37: 管理员可以导出族谱数据（支持迁移）
- FR38: 系统提供健康检查接口，用于检测核心依赖组件的连通性
- FR39: 系统支持容器化一键部署

## Non-Functional Requirements

### Performance

| 场景 | 目标 | 说明 |
|------|------|------|
| 首屏加载 | < 3s | 常规网络条件下 |
| 树图初始渲染 | < 2s | 千级节点规模 |
| 成员详情页 | < 1s | 含照片加载 |
| 批量导入 | 50 人 < 30s | 含校验与关系建立 |
| 关系网生成 | 千级 < 10s | 小规模同步模式 |

### Security

| 需求 | 说明 |
|------|------|
| 传输加密 | 生产环境使用 HTTPS |
| 存储加密 | 敏感字段按需加密存储 |
| 访问控制 | 按角色（管理员/成员）控制数据访问 |
| 操作审计 | 关键操作记录操作人、时间、内容 |
| 数据归属 | 数据存储在家族自有环境，支持备份与导出 |

### Scalability

| 维度 | MVP 目标 | 扩展预留 |
|------|----------|----------|
| 成员规模 | 万级 | 百万级 |
| 关系网更新 | 同步（千级） | 异步队列 + 增量更新 |
| 并发用户 | 单家族 < 50 人同时在线 | 按需扩展 |

### Accessibility

| 需求 | 说明 |
|------|------|
| 目标等级 | WCAG 2.1 AA |
| MVP | 键盘导航、焦点管理、基础语义化 |
| 后期 | 适老化（大字体、简化操作、长辈友好交互） |
