### 四、推荐方案（核心字段、关系实体、版本管理）

#### 1. 成员核心字段推荐

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `name` | 字符串 | 姓名（含曾用名、字、号等可拆分或扩展） |
| `gender` | 枚举 | 男/女/未知 |
| `birth_date` | 日期 | 出生日期（可为近似，如「约 1900 年」） |
| `death_date` | 日期 | 逝世日期，可空 |
| `birth_place` | 字符串 | 出生地/籍贯 |
| `biography` | 文本 | 生平 |
| `avatar` | 文件引用 | 照片/头像 |
| `generation` | 整数 | 世代（如第几世），便于排序与筛选 |
| `branch` | 字符串 | 房支/分支，可空 |
| `is_alive` | 布尔 | 是否在世（影响隐私与可见性） |
| `created_at` / `updated_at` | 时间戳 | 审计 |
| `extra` | JSONB | 扩展字段，支持自定义属性 |

**扩展字段（extra）示例：** 职业、学历、配偶姓名、迁居记录、重要事件等，可按家族习惯配置。

#### 2. 关系实体推荐

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 主键 |
| `genealogy_id` | 外键 | 所属族谱 |
| `person_from_id` | 外键 | 关系主体 |
| `person_to_id` | 外键 | 关系客体 |
| `relation_type` | 枚举 | 关系类型（见下表） |
| `order` | 整数 | 同类型多关系时的顺序（如多子女） |
| `start_date` / `end_date` | 日期 | 关系有效时间（如婚姻起止），可空 |
| `extra` | JSONB | 扩展（如过继原因、收养说明等） |

**关系类型（relation_type）建议：**

| 值 | 说明 | 方向 |
|----|------|------|
| `father` | 父亲 | person_from=父, person_to=子 |
| `mother` | 母亲 | person_from=母, person_to=子 |
| `spouse` | 配偶 | 双向，可多条（多配偶） |
| `son` / `daughter` | 子女 | 可由 father/mother 推导，也可显式存储 |
| `adopted` | 收养 | person_from=养父母, person_to=养子女 |
| `foster` | 过继 | person_from=过继方, person_to=被过继者 |

**说明：** 父子、母子、配偶为最基础关系；兄弟、堂兄弟等可由父子关系推导，也可视需求增加 `sibling` 等类型以加速查询。

#### 3. 族谱版本管理推荐

| 概念 | 说明 |
|------|------|
| **版本快照** | 每次「定稿/出版」时创建版本，保存当时成员与关系的快照或引用 |
| **版本表** | `id`, `genealogy_id`, `version_number`, `name`（如「2025 年续修版」）, `snapshot_at`, `created_at` |
| **快照策略** | 方案 A：冗余存储关键表快照（适合百万级，查询简单）；方案 B：仅存版本时间点 + 增量变更日志（省空间，需按时间回放） |
| **当前工作版** | 日常编辑针对「当前草稿」，定稿时生成新版本，草稿清空或进入下一轮编辑 |

**推荐：** 百万级规模下，版本表只存元数据；具体数据通过「版本时间点 + 软删除/状态」在成员表、关系表中标记，或采用「版本分支表」存储差异，避免全量快照占用过大。

#### 4. 审批相关表结构（PG）

| 表 | 用途 |
|----|------|
| `approval_flow` | 审批流程定义（如：成员生平修改 → 需族长审批） |
| `approval_request` | 单次审批请求（关联成员、修改内容、申请人、状态） |
| `approval_step` | 审批步骤（多级审批时使用） |
| `approval_record` | 审批记录（谁在何时通过/驳回，意见） |

### 五、百万级关系网更新策略建议

- **同步 vs 异步**：百万级建议**异步更新**，增删改成员后投递任务到队列，后台生成/更新关系网，避免接口超时。
- **增量更新**：若关系网可增量计算（如仅局部子树变化），优先增量；否则全量重算需控制频率与资源。
- **缓存/物化**：关系网结果可物化到表或缓存，前端按需加载分页或子图，避免每次实时计算。

### 十一、错误处理与异常（Phase 1 错误处理分支）

#### 1. API 错误响应

| 维度 | 决策 |
|------|------|
| 错误结构 | 需提供推荐方案 |
| 状态码 | 需提供规范 |
| 校验失败 | 需返回具体字段错误 |

**错误响应结构推荐：**

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数校验失败",
  "details": [
    { "field": "name", "message": "姓名不能为空" },
    { "field": "birth_date", "message": "出生日期格式错误，应为 YYYY-MM-DD" }
  ]
}
```

**HTTP 状态码规范推荐：**

| 状态码 | 场景 |
|--------|------|
| 400 | 请求参数错误（格式、类型） |
| 401 | 未认证 / Token 失效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 冲突（如关系冲突、重复提交） |
| 422 | 业务校验失败（如关系循环、逻辑错误） |
| 500 | 服务器内部错误 |

#### 2. 业务异常

| 场景 | 决策 |
|------|------|
| 关系冲突 | 需提供提示推荐 |
| 循环引用 | 需提供检测与拦截推荐 |
| 删除有关联的成员 | 不允许删除 |
| 成员状态标识 | 灰色 badge：已离世；红色 badge：已逐出/叛出家族 |

**关系冲突提示推荐：**  
「无法建立关系：A 与 B 已存在父子关系，不能同时建立反向关系。」或「该关系与现有关系冲突，请检查后重试。」

**循环引用检测与拦截推荐：**  
建立/修改关系时，从新关系出发做可达性检测（如 BFS/DFS），若存在 A→B→C→…→A 的环路则拒绝操作，并返回：「检测到关系循环（如 A→B→C→A），请修正后重试。」

### 十三、边界与异常场景（Phase 1 边界分支）

#### 1. 数据边界

| 场景 | 决策 |
|------|------|
| 同名同姓 | 使用出生日期辅助标识 |
| 出生日期不详 | 需提供存储与展示推荐 |
| 无子女/无父母 | 需提供边界处理推荐 |

**出生日期不详存储与展示推荐：**

| 存储 | 展示 |
|------|------|
| 精确：`birth_date = "1900-01-15"` | 「1900 年」或「1900 年 1 月」 |
| 约：`birth_date_approx = "约1900"`，`birth_date_precision = "year"` | 「约 1900 年」 |
| 朝代：`birth_era = "清末"`，存于 `extra` | 「清末」 |
| 不详：`birth_date = null`，`birth_date_known = false` | 「不详」 |

**无子女、无父母等边界推荐：**

| 场景 | 处理 |
|------|------|
| 无子女 | 允许，关系表中无对应 `person_to` 记录 |
| 无父母（始祖） | 允许，`person_from` 为空或指向虚拟「始祖」节点 |
| 配偶已故/未录入 | 配偶关系可单边存在，`person_to` 可空或为占位 |
| 过继/出继 | 可保留生父母关系 + 养父母关系，用 `relation_type` 区分 |

#### 2. 并发与一致性

| 场景 | 决策 |
|------|------|
| 多人同时编辑同一成员 | 不允许（编辑时加锁或占位，他人只读） |
| 关系网异步更新中 | 前端展示上一版结果，使用「计算中」遮罩 |

## Phase 3: SCAMPER Method（概念深化）

> 对核心功能与架构进行替代、组合、适配、修改、他用、消除、反转，挖掘改进空间。

---

### S - Substitute（替代）

| 原设计 | 替代方案 | 潜在收益 |
|--------|----------|----------|
| 成员生平审批需人工 | 引入 AI 辅助初审（敏感词、格式校验） | 减轻审批人负担，加快通过 |
| 站内信为主通知 | 默认微信/小程序推送，站内信为辅 | 提高触达率，长辈更易收到 |
| 关系网全量物化 | 热点分支物化 + 冷数据按需计算 | 降低存储与计算成本 |
| Excel 批量导入 | 支持 CSV、支持在线表格粘贴 | 降低使用门槛，快速试导入 |
| 编辑锁（独占） | 乐观锁 + 冲突时提示「他人已修改，请刷新」 | 减少误锁，体验更平滑 |

---

### C - Combine（组合）

| 组合对象 | 组合方式 | 潜在收益 |
|----------|----------|----------|
| 树状图 + 世系表 | 同一页面支持切换视图，共享选中状态 | 减少切换成本，上下文一致 |
| 审批 + 邀请 | 统一「待办中心」，审批与邀请审批同入口 | 管理员操作更集中 |
| 导入 + 导出 | 导入前可下载模板，导出格式与导入模板兼容 | 支持「导出→修改→再导入」闭环 |
| 纪念日提醒 + 家族公告 | 在「家族动态」中统一展示 | 信息聚合，减少打扰 |
| 成员详情 + 关系图 | 点击成员时侧边展示其局部关系子图 | 快速理解亲缘关系 |

---

### A - Adapt（适配）

| 原设计 | 适配方向 | 潜在收益 |
|--------|----------|----------|
| 仅支持中国家族关系 | 预留 `relation_type` 扩展，支持自定义关系类型 | 适配不同家族习惯（如少数民族、海外分支） |
| 富文本 + Markdown | 支持从 Word 粘贴并转 Markdown | 降低从现有文档迁移成本 |
| 按分支懒加载 | 支持「按房支」「按地区」「按世代」多种切分 | 适配不同家族组织结构 |
| 单一族谱 | 预留「族谱关联」概念，支持联姻家族互链 | 为跨家族场景留扩展 |

---

### M - Modify（修改）

| 原设计 | 修改方向 | 潜在收益 |
|--------|----------|----------|
| 关系网增删改后全量/大范围重算 | 仅重算受影响子树，其余复用缓存 | 百万级下显著缩短更新时间 |
| 成员状态仅灰/红 badge | 增加「待确认」「待审批」等中间状态 | 流程更清晰，减少歧义 |
| 出生日期不详用 null | 引入 `birth_date_precision`（year/month/day/unknown） | 支持「约 1900 年」「清末」等精细表达 |
| 审批通过后直接生效 | 支持「定时发布」或「版本发布」 | 便于修谱时批量上线 |

---

### P - Put to Other Uses（他用）

| 原能力 | 他用场景 | 潜在收益 |
|--------|----------|----------|
| 族谱数据 | 生成家族年报（人口统计、迁徙、大事记） | 增强家族凝聚力，可打印留念 |
| 关系网数据 | 导出为 GEDCOM 供其他软件使用 | 数据可迁移，生态兼容 |
| 成员生平 | 生成「家族故事集」电子书 | 文化传承，可分享给后代 |
| 审批流程 | 复用为「成员信息纠错提议」流程 | 众包纠错，减轻管理员负担 |

---

### E - Eliminate（消除）

| 可消除项 | 消除理由 | 替代或简化 |
|----------|----------|------------|
| 按在世/去世的可见性控制 | 已明确无需 | 直接移除，简化权限逻辑 |
| 敏感信息对部分成员隐藏 | 已明确无需 | 直接移除 |
| 多租户支持 | 单家族自用 | 不实现，降低复杂度 |
| 离线支持 | 已明确无需 | 不实现 |
| 全量关系网实时计算 | 百万级不可行 | 必须异步+物化 |

---

### R - Reverse（反转）

| 原设计 | 反转思路 | 潜在收益 |
|--------|----------|----------|
| 成员邀请需审批 | 管理员可「预录入成员」再批量发送邀请 | 适合修谱时集中录入后统一邀请 |
| 关系由用户建立 | 导入时根据「父亲姓名」等自动建立关系 | 减少手工建立，提高导入效率 |
| 树图从根向下展开 | 支持「从当前用户向上追溯」视图 | 满足「我是谁的后人」的视角 |
| 审批人固定 | 支持「审批人可委托」或「审批人缺席时自动转交」 | 避免审批卡住 |

---

### SCAMPER 提炼：建议采纳的改进

| 优先级 | 改进项 | 来源 |
|--------|--------|------|
| 高 | 热点分支物化 + 冷数据按需计算 | S |
| 高 | 树状图 + 世系表同页切换、共享选中 | C |
| 高 | 关系网增量更新（仅重算受影响子树） | M |
| 高 | 导入时根据父亲/母亲姓名自动建关系 | R |
| 中 | 统一待办中心（审批+邀请） | C |
| 中 | `birth_date_precision` 支持日期不详 | M |
| 中 | 支持「从当前用户向上追溯」视图 | R |
| 中 | 导入支持 CSV、在线粘贴 | S |
| 低 | 预留自定义关系类型 | A |
| 低 | 族谱年报、家族故事集 | P |

### 决策树 2：关系网技术选型

```
[关系网实现]
    │
    ├─ 数据规模 < 1 万？
    │       ├─ 是 → 同步计算 + 简单缓存
    │       │         └─ 增删改后直接重算，结果缓存 5–10 分钟
    │       │
    │       └─ 否 → 进入下一决策
    │
    ├─ 数据规模 1 万 – 10 万？
    │       ├─ 是 → 异步队列 + 全量物化
    │       │         └─ Celery 任务，增删改后投递，全量重算后更新物化表
    │       │
    │       └─ 否 → 进入下一决策
    │
    └─ 数据规模 > 10 万（含百万级）？
            ├─ 是 → 异步 + 增量更新 + 热点物化
            │         ├─ 仅重算受影响子树（增量）
            │         ├─ 热点分支（如当前用户所在支）优先物化
            │         └─ 冷数据按需计算或延迟物化
            │
            └─ 前端展示策略
                    ├─ 更新中 → 展示上一版 +「计算中」遮罩
                    └─ 更新完成 → 刷新或推送前端更新
```

### 决策树 3：导入流程决策

```
[用户发起批量导入]
    │
    ├─→ 选择文件类型
    │       ├─ Excel → 解析 + 校验
    │       ├─ CSV → 解析 + 校验
    │       └─ 在线粘贴 → 解析 + 校验
    │
    ├─→ 校验结果
    │       ├─ 全部通过 → 预览 + 确认导入
    │       ├─ 部分错误 → 错误报告 + 允许修正后重试
    │       └─ 全部失败 → 错误报告 + 提示检查模板
    │
    └─→ 确认导入后
            ├─ 成员匹配（父亲/母亲姓名）→ 自动建立关系

            └─ 导入完成 → 触发关系网更新任务
```

### 决策树 5：风险与应对

```
[潜在风险]
    │
    ├─ 关系网更新耗时过长
    │       ├─ 优化：增量更新、热点物化
    │       └─ 降级：允许「部分更新」或「延迟更新」模式
    │
    ├─ 导入数据质量差（重名、日期格式混乱）
    │       ├─ 预防：模板校验、格式规范、示例说明
    │       └─ 应对：错误报告 + 支持单条修正后重导
    │
    ├─ 审批流程卡住（审批人长期不处理）
    │       ├─ 预防：待办提醒、超时提醒
    │       └─ 应对：审批委托、自动转交、升级机制
    │
    └─ 百万级前端渲染卡顿
            ├─ 预防：懒加载、分片、虚拟滚动
            └─ 应对：限制单次展示节点数、强制分页/下钻
```
